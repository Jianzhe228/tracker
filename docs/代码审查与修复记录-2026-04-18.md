## 一、数据正确性修复

这一层的问题**会污染历史统计数据**（focus_sessions / task 完成历史），最优先处理。

### B1 · 休息模式暂停导致少休息

**位置**：`src/stores/timerStore.ts` tick() 函数 break 分支。

**问题**：focus 分支的 elapsed 计算是 `now - startedAt - accumulatedPauseMs - ongoingPauseMs`，而 break 分支只有 `now - startedAt`。用户在休息时按暂停、再恢复，暂停期被当作休息消耗，实际少休息。

**为什么一致性比对发现**：pause 按钮在 FocusModal.vue:369 对 running 状态一律可见，没有按 mode 限制。说明 pause 在 break 模式是被设计允许的，所以两条路径必须对称。

**修复**：把 focus 的减法算法搬到 break。**未选择**"禁用 break 模式 pause 按钮"——那会让一个**合法功能**消失，不是真正的修复。

---

### B2 · 重启把离线时间当成专注写入统计

**位置**：`src/stores/timerStore.ts:818-834` 原 `hydrateFromStorage` 的 running 分支。

**问题**：应用启动时若发现 `restored.status === 'running'` 且 `remainingSeconds ≤ 0`，直接调用 `finalizeFocusSession('completed')`。此时 segments 里 open 段的 `startedAt` 是离线前的旧值，**`spentSeconds` 会把整个离线期算成专注时间写入 `focus_sessions`**，污染统计图表。

**为什么不只是"加个离线时长判断"**：问题不是"应不应该 finalize"，而是"用户根本没在专注，系统擅自写数据"。正确语义是：**重启恢复应由用户决定**，UI 上本来就有 `promptRecovery` 流程。

**修复**：running 分支删除所有自动 finalize 代码，统一进入 `paused + awaitingRecovery`，让 `promptRecovery` 弹对话框由用户选择"继续"或"放弃"。放弃则 `resetToIdleFocus` 清干净；继续则 resume 继续。

---

### B3 · countup 模式 elapsedSeconds 被离线期污染

**位置**：同上 `hydrateFromStorage`。

**问题**：原代码 `elapsedSeconds.value += elapsed`（其中 elapsed 是 Date.now() - restored.lastTickAt，即离线时长）。如果用户在 promptRecovery 里选"继续"，elapsedSeconds 已经被污染，下一 tick 虽然会重算，但**期间的 UI 会显示被膨胀的时间**，且如果用户选"继续"后继续专注，累加到 focusSecondsToday 时也有错。

**修复**：彻底不加 elapsed。关键的 trick 是把离线时长累加到 `accumulatedPauseMs`——这样：
- `startedAt` 仍是旧值（不动）
- 离线期从 elapsed 计算里被自然扣除（公式：`now - startedAt - accumulatedPauseMs`）
- resume 后所有 segment 的 duration 自然正确

**为什么不改 startedAt**：因为 segments[].startedAt 与 pauseOffsetMs 有耦合，改 startedAt 必须同步调整所有 segment，风险高。累加 accumulatedPauseMs 只需一处改动，语义清晰。

---

### B4 · 番茄数计算硬编码 60 秒兜底

**位置**：`src/stores/timerStore.ts:559` `finalizeFocusSession`。

**问题**：`const focusSeconds = Math.max(60, totalSeconds.value);` 用 60 秒作为兜底——如果 `totalSeconds` 异常为 0，会把每分钟算作一个番茄，与用户设置（默认 25 分钟）严重脱钩。

**修复**：改用 `getDefaultSeconds('focus')`，从 `settingsStore.pomodoro.focusMinutes` 获取真实默认值。

---

### B5 · 重复任务 sibling 被强制标完成

**位置**：`src/stores/taskStore.ts` 原 `completeOverdueRecurringSiblings`（已删除）。

**问题**：用户完成今天的重复任务时，系统静默把所有**过期未完成**的同规则实例也改成 `done + completedAt=now`。这违反历史事实——"用户昨天没做"应该保留为 overdue/missed，而不是伪造成"昨天按时完成"。对周统计图、完成率图、趋势图都是数据污染。

**为什么选择直接删除而不是"弹窗询问"**：
1. 静默改写历史数据是错误的。任何需要"也把过期的标完成"的用户，可以**手动勾选**那些任务。
2. 保留行为需要加对话框、新交互、测试，复杂度高。
3. 删除后用户真正丢失的能力接近零：今天的任务仍然能完成。

**后续改进空间**：如果后续用户反馈确实需要批量操作，可以加一个**只在"过期"区域显示的工具条按钮**，显式批量标记。那是新功能，不是 bug 修复。

---

### B6 · toggleTask 全局单任务锁

**位置**：`src/stores/taskStore.ts:58, 295-300`。

**问题**：原实现 `togglingTaskId: number | null`，任意时刻只允许**一个**任务处于"切换中"。快速连续勾选多个任务时，第 2 个起立即返回 `{ ok: false, reason: 'sync_failed' }`，用户体感为"点不动"。

**修复**：改成 `togglingTaskIds: Set<number>`，按 id 独立加锁，允许不同任务并发，同一任务重入仍然拒绝（幂等）。

---

### B11 · 接受 AI 预测形成自反馈闭环

**位置**：`src/stores/predictionStore.ts:115`。

**问题**：`acceptPrediction` 调 `taskStore.addTask` 时没传 `skipPredictionRecord`。`addTask` 内部会 `recordTaskCreation`，也就是把 AI 建议的"新创建任务"写回 `task_creation_history`。下一轮 `task_history_analyzer` 会把自己的建议当成用户历史分析——**预测引擎在训练自己**，形成置信度越来越高的伪数据。

**修复**：一行参数 `skipPredictionRecord: true`。

## 二、边界逻辑修复

### B7 · updateTask 只拦截 done、忽略 cancelled

**位置**：`src/stores/taskStore.ts` `updateTask` 函数。

**问题**：只对 `payload.status === 'done'` 检查未完成子任务，阻止"父完成而子未完成"。但改成 `cancelled` 时没有任何处理，留下悬挂 todo 子任务（父已取消、子还在）。

**修复**：cancel 父任务时**级联**把子树中所有 todo 子任务也标 cancelled。
- 没有用"抛错阻止"——取消是用户意图，系统应该顺从。
- 没有级联 done 子任务——已完成的是事实，不应覆盖。
- Tauri 侧通过 `updateTaskCmd` 串行更新每个子任务（回滚用 snapshot）。

---

### B8 · copyTaskToToday 把已完成子任务复制成 todo

**位置**：`src/stores/taskStore.ts` `copyTaskToToday`。

**问题**：原实现复制所有子任务，但 `addTask` 默认 `status: 'todo'`。语义上相当于**把已完成子任务"复活"成今天的待办**——用户复制到今天是想继续做，而不是重做已经完成的。

**修复**：过滤 `status !== 'done' && status !== 'cancelled'`，与 `rescheduleToToday` 对齐（它本就这么做）。
测试里有一条 `copies a task with all its subtasks` 的原测试也相应重写为只复制未完成子任务。

---

### B9 · inbox 项目 ID 硬编码

**位置**：`src/utils/constants.ts`（新）、`src/stores/taskStore.ts:624`。

**问题**：项目被删除时，其任务 `projectId` 回落到 1（假设 1 是 inbox）。这个"1"散落在代码里，没有语义。若将来 schema 重做或导入数据后 inbox 的 id 不是 1，会静默破坏。

**修复**：新增 `INBOX_PROJECT_ID = 1`，替换生产代码处的字面量。**没有**替换测试里的 `projectId: 1`——那些是测试 fixture 的随意选择，不一定代表"inbox"语义，替换会错误地赋予含义。

---

### B12 · approveJob 部分失败静默

**位置**：`src/stores/aiStore.ts` `approveJob`。

**问题**：一个 action 执行失败只 `console.error`，然后继续执行下一个，最后不管成败都把 job 从 pending 移除。用户不知道有失败。

**修复**：
1. 失败 action 标 `status: 'failed'`（需同步扩展 `AiAction` 类型）。
2. 只有**没有 pending 剩余**时才移除 job（失败也算 resolved，但 pending 仍在则保留等用户重试）。
3. 通过 `uiStore.notify` 告知失败个数。

**为什么不抛异常**：一个 action 失败不代表其他也该中止。AI 给的是若干建议，允许部分采用。

## 三、交互与 UX

### U1 · FocusModal 缺少 focus trap

**位置**：`src/components/FocusModal.vue`。

**问题**：原本只监听 ESC。Tab 键按多了会跳出到背景页，违反 modal 的语义；关闭后焦点没有归位到触发按钮，依赖屏幕阅读器的用户会"迷路"。

**修复**：
1. 加 `modalRoot` ref 指向 dialog 根元素。
2. 打开时记录 `previousActive = document.activeElement`，关闭时还原。
3. 监听 Tab 键，按 focusables 第一/最后元素做循环。

**没有**为此引入 `vue-focus-trap` 一类的库——这个实现不到 30 行，依赖一个库只为了一个组件不值当。

---

### U3 · 启动瞬间弹多条截止通知

**位置**：`src/stores/taskStore.ts` `startDeadlineWatcher`。

**问题**：启动时立即调用 `checkDeadlines()`。如果今日有 3 个即将截止任务（已做合批），仍然在应用刚启动那一瞬间推送。这个时机用户还在适应 UI，提示很突兀。

**修复**：首次检查延迟 5 秒。此外在 timeout 里加了 `if (deadlineWatcherInterval)` 防御——避免用户关闭 watcher 后延迟回调仍然执行。

---

### U4 · predictionStore 在模块级自启动

**位置**：`src/stores/predictionStore.ts`，原模块底部 `void loadPendingPredictions(); void startListening();`。

**问题**：store 一被 import 就开始发起网络请求、注册 Tauri 事件监听。这种**import 副作用**有几个坏处：
1. 测试环境下也会执行，需要在每个测试里打 mock。
2. 用户关闭预测功能后无法真正停掉。
3. SSR / 非 Tauri 环境静默失败。

**修复**：抽出 `init()` 函数，由 `App.vue` 的 `onMounted` 显式调用，并加 `initialized` 幂等标记。

---

### U6 · 缺少统一的 EmptyState 组件

**位置**：新增 `src/components/ui/EmptyState.vue`。

**问题**：当前列表/图表空数据时各自自绘一段"暂无数据"文案，样式散乱。

**修复**：只建组件，**不强制替换现有实现**。这是一个渐进采纳的锚点——后续触到空状态时顺手替换即可。一次性重构 N 处显示只是增加 diff 噪音。

## 四、UI 视觉统一

### UI-1 · tailwind 设计 token

**位置**：`tailwind.config.js`（重写，原先是空的）。

**问题**：原配置 `theme.extend: {}` 什么都没有。项目里到处写 `bg-blue-600`、`bg-indigo-500`、`bg-sky-600`、`bg-violet-600`，四套主色轮流用——因为没有主色约定。

**修复**：扩展 `colors` 加 `primary` / `success` / `warning` / `danger` 四个语义色阶。**关键点**：没有改掉任何现有 class，只是增加了新别名。这意味着：
- 现存的 `bg-blue-600` 仍然工作。
- 新写代码可用 `bg-primary-600`。
- 渐进迁移，不产生大量 diff。

---

### UI-2 · 图表调色板

**位置**：新增 `src/components/charts/palette.ts`。

**问题**：图表里硬编码 `#3b82f6`、`#94a3b8`、`#22c55e` 等 hex 共出现 40+ 次。换主题就要改 40 个地方。

**修复**：抽到 `chartPalette` 常量和 `chartSeriesPalette` 数组。和 UI-1 一样**不强制替换**现有图表代码——新图表直接用，老图表触到时替换。

---

### UI-3 · 成功色 green ↔ emerald 二选一

**位置**：src 下所有 `.vue`，通过 sed 批量替换。

**问题**：`bg-green-500` 和 `bg-emerald-500` 并存，同一页面相邻元素可能用的是不同色族，虽然肉眼差别小，但是**设计系统是否有纪律**的信号。

**修复**：全量 `green → emerald`（bg / text / border / ring / hover: / focus-visible:ring / group-hover:bg 全类别）。选 emerald 是因为它与 Tailwind v3 的"新语义"一致（更现代的绿，对比度更好）。

替换后 `grep green-` 归零。

## 五、刻意跳过的事项

### B10 · prediction 通知按 id 精确匹配

**为什么跳过**：原实现 `.slice(0, createdCount)` 取 pending 列表前 N 条作为"新预测"——如果列表里有旧 pending 条目，会误推旧的。

**为什么没修**：修复需要改 Rust 侧 `refresh_predictions` 的返回签名，把 `createdCount: usize` 换成 `createdIds: Vec<i64>`。涉及跨语言迁移、DB 层 `last_insert_rowid` 收集、前端类型同步。**风险 > 收益**，保留为独立 PR。

**临时可接受**：现实中 pending 列表大部分时间较短（24h cooldown 会清理），误推概率低。

---

### U7 · AI 轮询改事件

**为什么跳过**：`aiStore.init` 里 10 秒轮询 `loadPendingJobs`。理想方案是后端 `emit('ai:job:updated')`，前端 `listen`。同样涉及 Rust 改动，保守。

**临时可接受**：10 秒一次 SQLite 读没有实际性能问题。
