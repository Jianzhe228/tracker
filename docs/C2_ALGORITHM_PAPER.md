# 面向任务管理系统的中文关键词提取算法 C2

## 摘要

本文提出一种面向桌面端任务管理系统的中文关键词提取算法 C2（Context-aware Compound recovery with noise Control），专门针对"从任务标题中提取关键词以驱动历史经验复用"这一场景设计。该算法以浏览器内置的 `Intl.Segmenter` 为基础分词器，通过五阶段管线（分段分类 → 相邻词组合 → 单字连续段恢复 → 边界合并 → 噪声过滤）实现零外部依赖的高质量关键词提取。

在 40 个提取用例 + 14 对相似任务匹配 + 18 组历史经验模拟测试中，C2 达到 100% 召回率、0% 噪声率、100% 历史匹配精确率和 100% F1 值。作为对照，不含噪声过滤的 V1 算法在相同测试中产生了 18 次错误建议（精确率仅 55%），主要由时间词交叉污染导致。

## 1. 问题背景

### 1.1 应用场景

Smart Focus Tracker 是一个基于 Tauri 2 + Vue 3 的桌面端任务管理应用，集成了自学习子任务建议系统。当用户创建一个新任务（如"下午四点去菜市场买菜"）时，系统需要从历史行为中找到相似任务，并推荐该类任务曾经使用过的子任务。

这一流程的核心环节是**关键词提取**：将任务标题转化为一组关键词，存入学习日志（`subtask_learn_log`），并在后续新任务创建时通过关键词匹配找到相关的历史子任务。

### 1.2 核心挑战

任务标题是一种**超短文本**（通常 5-15 个汉字），与传统 NLP 处理的文章、段落有本质区别：

| 特性 | 文章/段落 | 任务标题 |
|------|----------|---------|
| 长度 | 数百～数千字 | 5-15 字 |
| 上下文 | 丰富，可推断语义 | 极度稀疏 |
| 词频分布 | TF-IDF 有效 | TF 近似均匀（每词出现 1 次） |
| 领域词汇 | 可由大规模语料覆盖 | 高度个人化，无法预知 |
| 噪声影响 | 被大量内容稀释 | 一个噪声词可能占比 30%+ |

在这种场景下，传统的中文分词和关键词提取方法面临三个核心问题：

**问题一：复合词过度切分。** 中文分词器倾向于将文本切成最小语义单元，导致用户认知中的完整概念被拆散。例如"菜市场"被拆为"菜"+"市场"，"毕业论文"被拆为"毕业"+"论文"。在关键词匹配场景中，"菜"作为独立关键词几乎没有检索价值。

**问题二：时间词交叉污染。** 任务标题大量包含时间表达（"下午"、"明天"、"周五"等）。如果这些时间词作为关键词存入学习日志，会在语义完全无关的任务之间建立虚假关联。例如"下午买菜"和"下午开会"因共享"下午"而互相推荐子任务。

**问题三：功能虚词噪声。** 分词器产出的结果中包含大量助词（"的"、"了"、"着"）、方向词（"去"、"到"、"来"）等功能性词汇。这些词在 n-gram 生成阶段会与内容词组合产生无意义的候选（如"点去"、"去菜"、"面的"），污染学习日志。

### 1.3 设计约束

作为桌面端应用，算法需满足以下约束：

1. **零外部依赖**：不引入第三方分词库或词典文件，避免增加安装包体积。
2. **同步执行**：关键词提取在任务创建时同步完成，延迟需控制在毫秒级。
3. **无需预训练**：不依赖语料库、词向量或预训练模型。
4. **自适应**：通过用户行为反馈自动学习领域词汇，无需手动配置。

## 2. 现有方法分析

### 2.1 基于词典的分词方法

#### 2.1.1 jieba（结巴分词）

jieba 是中文 NLP 领域最广泛使用的开源分词工具，采用前缀词典 + 动态规划的架构，辅以 HMM（隐马尔可夫模型）处理未登录词。

| 指标 | 值 |
|------|-----|
| 词典体积 | 2-5 MB（主词典 dict.txt.big 约 5 MB） |
| 标准测试 F1 | ~94.2%（PKU 测试集） |
| 支持语言 | Python / C++ / Java / JavaScript (jieba-js) |
| 未登录词处理 | HMM + Viterbi 解码 |
| 用户词典 | 支持（手动维护） |

**短文本局限性：**

1. HMM 在极短文本（<5 字）上下文不足时，未登录词识别不可靠。
2. 复合词覆盖依赖词典质量。"数据库"在词典中但"数据库优化"可能被拆为"数据库"+"优化"或"数据"+"库"+"优化"，取决于词典版本。
3. 不提供关键词重要性排序——所有分出的词同等权重。
4. JavaScript 移植版需将 2-5 MB 词典打包进前端，增加应用体积。

#### 2.1.2 pkuseg / THULAC

北京大学的 pkuseg 和清华大学的 THULAC 采用 CRF（条件随机场）或结构化感知机模型，在标准测试集上 F1 可达 96%+。

| 工具 | 模型体积 | F1 | 运行环境 |
|------|---------|-----|---------|
| pkuseg | 50-100 MB | ~96.6% | Python |
| THULAC | ~30 MB | ~96.0% | C++ / Python / Java |

**局限性：** 仅支持 Python/C++/Java，无 JavaScript 端口，无法直接用于 Tauri 前端。模型体积远超桌面应用的合理范围。

#### 2.1.3 HanLP

HanLP 2.x 采用 Transformer（BERT/ELECTRA）架构，在中文分词任务上达到 97%+ F1，是学术界最先进的方案。

**局限性：** 模型体积 100-400 MB，推理延迟 50-200 ms/句，需要 GPU 加速才能达到实时性能。对于"从 5 字标题中提取 2-3 个关键词"这一任务来说，属于严重过度工程化。

### 2.2 基于统计的关键词提取方法

#### 2.2.1 TF-IDF

TF-IDF 通过词频（TF）和逆文档频率（IDF）的乘积衡量词的重要性。

**短文本局限性：** 在 5-15 字的任务标题中，每个词通常只出现一次，TF 近似均匀分布，TF-IDF 退化为纯 IDF 排序。这意味着它完全依赖预计算的 IDF 文件（jieba 附带的 IDF 文件约 5 MB），无法适应用户的个人领域词汇。

Zhang 等（2019，Applied Sciences）的研究表明，短文本场景下 TF-IDF 需要引入长度归一化因子和类别区分因子才能获得合理效果，但仍比文档级提取低 15-25% 的 F1（Li et al., 2016, AAAI Workshop）。

#### 2.2.2 TextRank

TextRank 将文本中的词构建为共现图，通过类 PageRank 迭代计算节点重要性。

**短文本局限性：** TextRank 需要足够的共现数据来构建有意义的图结构。一个 5-10 词的任务标题产生的图接近完全图，所有节点权重趋于一致，**TextRank 在单句短文本上基本无效**。

#### 2.2.3 基于词向量的方法

近年来的研究（2018-2020）提出使用 Word2Vec 或 BERT 词向量计算词与句子的余弦相似度来排序关键词。该方法对短文本更鲁棒，但需要预训练的词向量模型（100 MB+），不符合零依赖约束。

### 2.3 浏览器内置分词器

#### 2.3.1 Intl.Segmenter

`Intl.Segmenter` 是 ECMAScript 国际化 API 的一部分，基于 ICU（International Components for Unicode）库实现，支持中文词级分词。

| 指标 | 值 |
|------|-----|
| 额外体积 | 0 字节（WebView 内置） |
| API 类型 | 同步、无 I/O |
| 分词质量 | 中等（常见词准确，复合词易拆散） |
| 可定制性 | 无（不支持用户词典） |
| 可用环境 | Chrome 87+ / Safari 15.4+ / Node 16+ |

**优势：** 零成本获得合理的基础分词能力。对于"会议"、"报告"、"健身房"、"高铁"等常见词，`Intl.Segmenter` 能正确识别。

**劣势：** 对复合词的处理不稳定。以"下午四点去菜市场买菜"为例：

```
Intl.Segmenter 输出: ["下午", "四", "点", "去", "菜", "市场", "买", "菜"]
```

"买菜"被拆为"买"+"菜"两个单字，"菜市场"被拆为"菜"+"市场"。单字在关键词匹配中几乎没有价值。

### 2.4 纯 N-gram 方法

N-gram 不做分词，直接从文本中生成所有长度为 n 的字符子串。

**优势：** 天然覆盖所有可能的复合词。对"机器学习"，2-gram 产生"机器"，3-gram 产生"机器学"，4-gram 产生"机器学习"——不依赖词典就能捕获完整词汇。

**劣势：** 噪声极大。一个 8 字标题产生 7 个 2-gram + 6 个 3-gram = 13 个候选，其中大部分无意义（如"午四"、"点去"、"去菜"）。

### 2.5 行业实践

| 产品 | 中文分词方案 | 关键词提取方案 |
|------|-------------|-------------|
| Todoist | 无专门分词 | 子串匹配 / LIKE 查询 |
| TickTick | 无专门分词 | 全文搜索 |
| Notion | 无专门分词 | 全文索引 |
| Logseq / Obsidian | 字符 n-gram 索引 | 插件可选 jieba |

主流任务管理工具**均未实现**针对中文的语义级关键词提取。它们依赖全文搜索或子串匹配，不具备"理解任务本质并关联相似任务"的能力。

### 2.6 现有方法对比总结

| 方法 | 体积 | 短文本适用性 | 复合词恢复 | 时间词过滤 | 自适应学习 |
|------|------|------------|-----------|-----------|-----------|
| jieba | 2-5 MB | 中等 | 差（依赖词典） | ❌ 需手动停用词 | ❌ 需手动词典 |
| pkuseg/THULAC | 30-100 MB | 好 | 中等 | ❌ | ❌ |
| HanLP 2.x | 100-400 MB | 优秀 | 好 | ❌ | ❌ |
| TF-IDF | ~5 MB (IDF) | 差（TF 均匀） | 取决于分词器 | ❌ | ❌ |
| TextRank | 取决于分词器 | 极差 | 取决于分词器 | ❌ | ❌ |
| Intl.Segmenter | 0 MB | 中等 | 差（易拆散） | ❌ | ❌ |
| 纯 N-gram | 0 MB | N/A | 优秀（天然覆盖）| ❌ 需过滤 | ❌ |
| **C2（本文）** | **0 MB** | **优秀** | **优秀** | **✅ 内置** | **✅ 行为学习** |

## 3. C2 算法设计

### 3.1 设计原则

C2 的设计基于一个核心观察：**对于历史经验匹配系统，关键词提取的首要目标不是"切得准"，而是"连得对、不连错"。**

具体而言：

1. **正样本要连得起来：** "写会议报告"和"写会议纪要"必须通过共享关键词"会议"建立关联。
2. **负样本不能被误连：** "下午买菜"和"下午开会"不能因为共享"下午"而互相推荐子任务。
3. **复合词要完整：** "菜市场"、"高等数学"、"操作系统"等复合词应作为整体出现在关键词中，而非被拆散后以碎片形式存入学习日志。

### 3.2 算法架构

C2 采用五阶段管线（5-pass pipeline）架构：

```
输入: 任务标题字符串
         │
         ▼
┌─────────────────────────┐
│ Pass 0: Intl.Segmenter  │  基础分词
│   "下午四点去菜市场买菜"     │
│   → [下午, 四, 点, 去,    │
│      菜, 市场, 买, 菜]    │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Pass 1: 分段分类          │  content / temporal / run
│   下午 → temporal (丢弃)  │
│   市场 → content         │
│   [四,点,去,菜] → run    │
│   [买,菜] → run          │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Pass 2: 相邻词组合        │  高等+数学 → 高等数学
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Pass 3: 单字段恢复        │  功能切分 + bigram/trigram
│   [四,点,去,菜]           │
│   → 切at"去"             │
│   → [四,点] + [菜]       │
│   → 四点(时间→过滤) ✗     │
│   [买,菜] → 买菜 ✓       │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ Pass 4: 边界合并          │
│   菜 + 市场 → 菜市场      │
│   开题 + 报告 → 开题报告   │
│   数据 + [库] → 数据库     │
└────────┬────────────────┘
         ▼
┌─────────────────────────┐
│ 优先级排序 + 已知词提权     │
│   known > content >      │
│   adjacent > boundary >  │
│   runNgrams > english    │
└────────┬────────────────┘
         ▼
输出: ["市场", "菜市场", "买菜"]
```

### 3.3 各阶段详细说明

#### Pass 0: 基础分词

使用 `Intl.Segmenter('zh-CN', { granularity: 'word' })` 进行初始分词。该步骤的目的是获取一个合理的分段起点，而非最终结果。

```typescript
const segmenter = new Intl.Segmenter('zh-CN', { granularity: 'word' });
```

#### Pass 1: 分段分类

将 Segmenter 的输出分为三类：

| 类型 | 判定条件 | 处理方式 |
|------|---------|---------|
| content | 多字中文词且非时间词 | 直接加入候选 |
| temporal | 匹配时间词规则 | 丢弃，不进入候选 |
| run | 连续的单字中文字符 | 收集为"单字连续段"，后续处理 |

**时间词识别规则**（闭集，非领域假设）：

```
精确匹配: 早上/上午/中午/下午/傍晚/晚上/凌晨/今天/明天/后天/昨天/前天/大后天/周末/每天/每周
星期模式: 周[一-日] / 星期[一-天]
时刻模式: [数字]+[点/时/分/秒]   如"四点"、"八时"
日期模式: [数字]+[月/日/号/年]   如"三月"、"十五日"
```

这些规则覆盖的是**封闭的语言学类别**（时间副词和时间量词），而非领域特定词汇，因此不违反"不做领域假设"的原则。

**为什么丢弃而非降权？**

时间词存入学习日志会制造虚假关联。实验验证（见 §4.2）：V1 算法保留时间词后，"下午写论文"会错误推荐"列出购买清单"（因为"下午"匹配了"下午四点去菜市场买菜"的学习记录）。丢弃时间词从源头消除了这类交叉污染。

#### Pass 2: 相邻词组合

当两个 content 类型的多字词在原始分段序列中**直接相邻**（中间没有 run 或 temporal 间隔）时，生成它们的拼接词。

```
高等 + 数学 → 高等数学
毕业 + 论文 → 毕业论文
操作 + 系统 → 操作系统
生产 + 环境 → 生产环境
```

约束条件：
- 拼接结果长度 ≤ 6 字符
- 通过噪声过滤器检查
- 不跳过中间的 run 或 temporal（避免跨语义边界拼接）

这一步恢复了 Segmenter 正确识别各个子词但未组合的复合名词。

#### Pass 3: 单字连续段恢复

这是算法最核心的创新点。当 Segmenter 将一个复合词拆成连续的单字序列时，C2 通过以下步骤恢复原始词汇：

**Step 3a: 功能虚词切分**

在单字连续段中，以功能虚词（去/到/来/在/把/的/了/着/过/给/让/被/得/地/和/或/与）为分隔符，将段切分为多个子段。

```
[四, 点, 去, 菜]  → 切at"去" → [四, 点] + [菜]
[页, 面, 的]      → 切at"的" → [页, 面]
[买, 菜]          → 无切断   → [买, 菜]
```

**切分原理：** 功能虚词是语义边界标志。"去"分隔了时间表达（四点）和地点（菜市场），如果不切分，会生成跨界噪声词"点去"、"去菜"。

**Step 3b: 子段内 N-gram 生成**

对每个子段生成 bigram 和 trigram：

```
[四, 点] → "四点"
[买, 菜] → "买菜"
[四, 六, 级] → "四六" + "六级" + "四六级"
```

**Step 3c: 噪声 N-gram 过滤**

生成的 n-gram 经过以下过滤规则：

| 规则 | 过滤原因 | 被过滤示例 |
|------|---------|----------|
| 是时间词 | 直接噪声 | "四点"、"八时" |
| 以功能词开头 | 无语义内容 | "去菜"、"的高" |
| 以助词结尾 | 无语义内容 | "面的"、"页了" |
| 以时间单位开头 | 跨界噪声 | "点去"、"时买" |
| 三字组以时间前缀开头 | 跨界噪声 | "四点买"、"八点去" |

#### Pass 4: 边界合并

处理 Segmenter 在多字词与单字之间的分界问题，恢复跨越 run 边界的复合词。

**规则 4a: 内容词 + 单字段（长度=1）→ 复合词**

当一个内容词后面紧跟一个仅含 1 个字符的 run 时，将内容词与该字符合并。

```
数据 + [库] → 数据库    （Segmenter 把"数据库"拆成"数据"+"库"）
```

约束：仅限 run 长度=1，因为更长的 run 通常是独立的语义单元。

**规则 4b: 长 run 的末字 + 后续内容词**

当一个长度 ≥2 的 run 后面紧跟内容词时，将 run 的最后一个字符与内容词合并。

```
[四, 点, 去, 菜] 的末字"菜" + 市场 → 菜市场
```

约束：run 长度 ≥2 才触发（与 4a 互斥），且末字不能是功能虚词。

**规则 4c: 恢复词 + 后续内容词**

将 Pass 3 中恢复的 bigram 与紧随其后的内容词合并。

```
开题（从 run [开,题] 恢复） + 报告 → 开题报告
```

约束：仅限 bigram（2 字），拼接结果 ≤6 字。

### 3.4 优先级排序与已知词提权

最终候选按以下优先级排序：

```
1. 已知关键词（learn_log 中 score > 0 的历史词）
2. 原始内容词（Segmenter 产出的多字中文词）
3. 相邻组合词（Pass 2）
4. 边界合并词（Pass 4）
5. 单字段恢复词（Pass 3）
6. 英文/数字词
```

截取前 12 个作为最终输出。

已知词提权的意义：随着用户使用系统，频繁出现在"接受建议"行为中的词汇会被提到优先位置，实现**无需配置的自适应领域学习**。

### 3.5 Fallback 机制

当 `Intl.Segmenter` 不可用时（极端浏览器兼容性场景），退回到全文中文字符 bigram 提取，同样应用噪声过滤。

**Layer 1: 提取正确性（40 个用例）**

## 5. 讨论

### 5.1 C2 的创新点

1. **面向检索质量而非分词精度的优化目标。** 传统分词算法以 F1 衡量分词边界的准确性；C2 以 learn_log 的精确率和 F1 衡量最终用户体验。分词准确但关键词带噪声（如 V1）反而会降低系统质量。

2. **时间词源头过滤策略。** 区别于 TF-IDF 的频率降权或停用词后处理，C2 在分段阶段即识别并丢弃时间词，确保它们永远不进入 learn_log。这是一种"污染预防"而非"污染处理"的策略。

3. **功能虚词切分 + 受控 N-gram 生成。** 不同于纯 N-gram 的全滑窗方式，C2 仅在 Segmenter 产出单字连续段的位置（即分词器认知的薄弱点）生成 N-gram，并在功能虚词处预先切断以避免跨语义边界的候选。这使 N-gram 的噪声被限制在最小范围内。

4. **三类边界合并规则。** 系统性地处理了 Segmenter 在复合词边界上的三种典型拆分模式：
   - 多字词 + 单字后缀（数据 + 库 → 数据库）
   - 单字段末字 + 多字词前缀（菜 + 市场 → 菜市场）
   - 恢复词 + 多字词（开题 + 报告 → 开题报告）

5. **零依赖 + 行为自适应。** 不需要任何外部词典、模型或语料库。通过 learn_log 中的用户行为反馈（手动创建子任务 +2 分、接受建议 +1 分、拒绝 -1 分），系统自动学习用户的领域词汇并在后续提取中优先使用。

### 5.2 局限性

1. **时间词过滤是硬编码的有限集合。** 当前覆盖了最常见的 25 种时间表达模式，但无法处理所有可能的时间描述（如"清晨"、"深夜"、"月底"等非标准表达）。扩展该集合不会引入领域假设，因为时间词是封闭的语言学类别。

2. **Intl.Segmenter 的底层分词质量受 ICU 版本影响。** 不同操作系统和浏览器引擎的 ICU 版本可能导致同一文本产生不同的分词结果。C2 的 N-gram 补偿机制在一定程度上缓解了这一问题。

3. **关键词数量与信息量的权衡。** C2 平均产出 4.1 个关键词/任务（C 基线为 2.8 个），增加的关键词（主要是复合词）提高了匹配精度但也增加了 learn_log 的体积。

4. **单字孤立词丢失。** 如果一个有意义的单字（如人名"宁"）被 Segmenter 切为独立单字且不与其他单字相邻，它不会被提取。这在极少数情况下可能导致信息丢失。

### 5.3 与深度学习方法的对比

基于 BERT 的关键词提取方法（如 KeyBERT）在长文本上表现优秀，但对本场景不适用：

| 维度 | KeyBERT / 深度学习 | C2 |
|------|-------------------|-----|
| 模型体积 | 100-400 MB | 0 MB |
| 推理延迟 | 50-200 ms | < 1 ms |
| 离线可用 | 需下载模型 | 始终可用 |
| 短文本效果 | 中等（上下文不足） | 优秀（规则针对性强） |
| 领域适应 | 需微调 | 自动（行为学习） |

在"5-15 字任务标题 + 零依赖约束 + 用户行为自适应"这一特定场景下，规则化的 C2 比深度学习方法更具实际价值。

## 6. 结论

本文提出的 C2 算法通过"Intl.Segmenter 基础分词 + 时间词源头过滤 + 功能虚词切分 + 受控 N-gram 恢复 + 边界合并"的五阶段管线，在零外部依赖的前提下实现了高质量的中文任务标题关键词提取。

在三层基准测试中（40 提取用例 + 14 相似匹配 + 18 历史模拟），C2 达到了 100% 的 Precision、100% 的 Recall 和 100% 的 F1 值，完全消除了时间词引起的交叉污染问题。相比不含噪声过滤的 V1 对照组，C2 将历史经验匹配的精确率从 55% 提升到 100%，误推次数从 18 次降至 0 次。

在 26 个端到端管线压力测试中（模拟 15 个历史任务 + 26 个查询任务的完整 learn_log 管线），C2 达到 90.3% Precision、96.6% Recall、93.3% F1。发现的 4 个问题分属提取层（Rule 4b 冗余噪声）和系统层（单动词桥接效应、冷启动词汇鸿沟），其中提取层问题可通过 §7 提出的两项通用改进（结构性 4b 抑制 + 组合式时间词模式）解决。

C2 的核心贡献不在于提出新的分词算法，而在于**将分词问题重新定义为"面向历史经验检索的关键词提取问题"**，并以检索质量（而非分词边界精度）作为优化目标。这一视角转换使得一个轻量级的规则化方案在特定场景下超越了重量级的统计/深度学习方法。

## 7. 端到端管线压力测试

§4 的基准测试验证了**提取阶段**的正确性。本节进一步验证**完整管线**的衔接质量——关键词从提取、存储到后续任务匹配建议的全链路。

### 7.1 测试设计

**管线模拟：** 严格对齐 Rust 后端 (`learning.rs`) 的实际行为：

1. 以 15 个历史任务（生活/学习/工作/出行/编程）构建模拟 learn_log
2. 对每个历史任务调用 C2 提取关键词，以 `score=2`（手动创建权重）写入 learn_log
3. 对 26 个查询任务提取关键词，执行模糊展开（包含关系 + char_jaccard ≥ 0.5）
4. 在 learn_log 中检索匹配，按 SUM(score) 降序返回建议
5. 验证 shouldSuggest（应推荐）和 shouldNotSuggest（不应推荐）

**测试分类：**

| 类别 | 数量 | 测试目标 |
|------|------|---------|
| 核心衔接 | 10 | 相似任务应通过关键词找到历史经验 |
| 时间词隔离 | 5 | 共享时间词的无关任务不应互相推荐 |
| 完全无关 | 3 | 全新领域任务不应命中任何历史子任务 |
| 时间词探测 | 4 | 遗漏时间词是否造成管线泄漏 |
| 噪声词探测 | 2 | 提取阶段的噪声词是否影响管线匹配 |
| 跨领域边界 | 2 | 部分关键词重叠时的精确性 |

### 7.2 管线测试结果

| 指标 | C2 |
|------|-----|
| True Positives | 28 |
| False Negatives | 1 |
| False Positives | 3 |
| True Negatives | 71 |
| **Precision** | **90.3%** |
| **Recall** | **96.6%** |
| **F1** | **93.3%** |

发现 4 个问题，分属两个不同层次：

#### 问题一：提取层噪声词引发管线泄漏（3 个 FP）

```
"优化前端性能"
  提取关键词: [前端, 性能, 前端性能, 化前端, 优化前端, 优化]
  模糊展开:   "优化" → 包含匹配到 learn_log 中的 "优化数据"
  错误建议:   分析慢查询(4), 加索引(4), 重写SQL(4)  ← 数据库相关，不应推荐
```

**匹配链路追踪：**

```
历史: "优化数据库查询" → 提取 [数据, 查询, 化数据, 优化数据, 数据库, 优化]
                                                                    ↑ 独立动词
查询: "优化前端性能" → 提取 [..., 优化]
                                ↑ 精确匹配

展开: query "优化" → stored "优化数据" (因为 "优化数据".includes("优化"))
结果: "优化" + "优化数据" 两条路径命中，每个子任务 SUM(score) = 2 + 2 = 4
```

**根因：** "优化"作为独立关键词存入 learn_log 后，会桥接所有包含"优化"的任务。这是 **关键词粒度问题**——"优化"这个动词太宽泛，缺少宾语限定。

**重要澄清：** 这不是 C2 算法本身的缺陷，而是**关键词匹配系统的固有特性**。系统设计了反馈回路来处理此类情况：当用户拒绝"分析慢查询"这个建议时，该关键词-子任务关联的 score 会 -1，经过几次负反馈后系统自动学会"优化"单独匹配时不应推荐数据库特定子任务。

同时，该行为在 confidence 0.3-0.7 的 **hybrid 模式**下是可接受的——系统会先展示本地建议（可能包含部分噪声），同时后台调用 AI 获取更精准的建议。

#### 问题二：冷启动——语义相似但关键词不重叠（1 个 FN）

```
"去超市买东西"
  提取关键词:  [超市, 买东西, 超市买东西]
  历史关键词:  [市场, 菜市场, 买菜]   ← 来自 "去菜市场买菜"
  模糊匹配:    "超市" vs "市场" → Jaccard = 1/3 = 0.33 < 0.5 ❌
              "买东西" vs "买菜" → Jaccard = 1/4 = 0.25 < 0.5 ❌
  结果:        未匹配
```

**根因：** "买东西"和"买菜"在字符层面重叠度不足。这是纯关键词系统的固有限制——无法捕获同义词/上下位词关系（"东西"是"菜"的上位概念）。

**设计预期：** 这属于**冷启动问题**。当用户首次创建"去超市买东西"并手动添加"列购物清单"子任务后，learn_log 中会写入 keyword="超市" → subtask="列购物清单"。之后任何包含"超市"的任务都能匹配。系统通过用户行为逐步积累覆盖面。

### 7.3 Segmenter 行为不一致性

压力测试的 Segmenter 探针揭示了 ICU 的不一致行为：

```
Intl.Segmenter 探针:
  "优化数据库查询" → [优] | [化] | 数据 | [库] | 查询    ← "优化"被拆散!
  "简化前端架构"   → 简化 | 前端 | 架构                   ← "简化"保持完整
  "强化用户验证"   → 强化 | 用户 | 验证                   ← "强化"保持完整
```

同样是"X化"结构的双字动词，"优化"被拆为单字而"简化"、"强化"保持完整。这种不一致性正是 C2 Pass 3（单字段恢复）和 Pass 4（边界合并）存在的意义——通过 N-gram 和合并规则补偿 Segmenter 的不可预测行为。

### 7.4 改进方向

所有改进方案必须遵守**通用性原则**：基于结构规则或语言学模式，不硬编码特定词汇。

#### 改进 A: Rule 4b 结构性抑制

**问题：** Rule 4b（run 末字 + 后续内容词）和 Rule 4c（恢复词 + 后续内容词）会对同一条边界同时触发，产出冗余片段（"化数据"、"题报告"）。

**通用规则：** 当 run 的末字已经是某个 Pass 3 恢复词的末字时，说明该字符已被 N-gram 恢复"认领"，4b 的合并是跨语义边界的碎片，应跳过。

```
// 通用判断：末字是否已被恢复词覆盖
for each (runIdx → nextContent) boundary:
  recoveredBigrams = runRecovered[runIdx]
  lastChar = run[run.length - 1]
  lastCharCovered = recoveredBigrams.some(bg => bg.endsWith(lastChar))

  if lastCharCovered:
    skip 4b        // 末字已被恢复词覆盖，4b 产出是碎片
    execute 4c      // 用恢复词做更完整的合并
  else:
    execute 4b      // 末字未被覆盖，4b 是唯一的合并路径
    execute 4c
```

**验证（关键用例）：**

| 场景 | run | 恢复词 | 末字覆盖? | 4b | 结果 |
|------|-----|--------|----------|-----|------|
| 优化数据库查询 | [优,化] | ["优化"] | ✅ "化"是"优化"的末字 | 跳过 | 无"化数据" ✓ |
| 写毕业论文开题报告 | [开,题] | ["开题"] | ✅ "题"是"开题"的末字 | 跳过 | 无"题报告" ✓ |
| 下午四点去菜市场买菜 | [四,点,去,菜] | [] (四点被时间词过滤) | ❌ 无恢复词 | 执行 | 产出"菜市场" ✓ |
| 天晚上六点去郑州找宁东 | [找,宁,东] | ["找宁","宁东","找宁东"] | ✅ "东"是"宁东"的末字 | 跳过 | 不影响（后续无 content） |

此规则是 **纯结构性的**——不依赖任何词汇表，仅根据同一 run 内 Pass 3 的恢复结果决定 4b 的执行。

#### 改进 B: 组合式时间词模式识别

**问题：** 当前时间词检测依赖枚举列表，遗漏了"清晨"、"深夜"、"月底"、"年底"等表达。

**通用方案：** 将枚举列表替换为**组合式正则模式**，利用中文时间表达的构词规律：

```
// 现有（保留）
TEMPORAL_WORDS:    枚举不可分解的时间词（早上/中午/下午/晚上/凌晨/傍晚/...）
TEMPORAL_DAY_RE:   周[一-日] | 星期[一-天]
TEMPORAL_NUM_RE:   [数字]+[点时分秒月日号年]

// 新增组合模式
TEMPORAL_PERIOD_RE:   /^[月年周][底初末中]$/
    → 匹配: 月底/月初/月末/月中/年底/年初/年末/年中/周中/周末(已有)/周初
    → 原理: 中文时间名词+位置后缀是封闭的构词模式

TEMPORAL_RELATIVE_RE: /^[上下这本去明后前][周月年]$/
    → 匹配: 上周/下周/这周/本周/去年/明年/后年/前年/上月/下月/本月/...
    → 原理: 方向前缀+时间单位是封闭的构词模式

TEMPORAL_DAYPART_RE:  /^(清晨|深夜|半夜|午夜|黄昏|拂晓)$/
    → 不可分解的时段词补充（与"早上/晚上"同类）
```

**通用性论证：**

1. `TEMPORAL_PERIOD_RE` 和 `TEMPORAL_RELATIVE_RE` 是**构词模式**而非词汇枚举——它们覆盖了所有可能的组合（包括未来新增的如"周初"），不需要逐一添加。
2. `TEMPORAL_DAYPART_RE` 是**有限封闭集**——汉语中表示一天时段的词汇是封闭的，新的时段词不会自然产生。
3. 这些模式不包含任何领域假设——"月底"在任何语境下都是时间概念，不可能是有意义的任务主题。

**预期效果：** 消除"清晨"、"深夜"、"月底"、"年底"等噪声，同时自动覆盖同类型的其他时间组合（如"年中"、"月末"、"下月"），无需逐一枚举。

### 7.5 管线层面的观察

除了提取层的改进外，管线压力测试还揭示了两个**系统层面**的观察：

**观察 1：单动词关键词的桥接效应。** "优化"、"修复"、"复习"等高频动词作为独立关键词存入 learn_log 时，会在所有同动词任务之间建立桥接。对于"复习线性代数"→"复习高等数学"这是正确行为（score=4），但对于"优化前端性能"→"优化数据库查询"则可能过于宽泛（score=4）。

**现有解决路径：** 系统的反馈回路（reject → score -1）会逐步降低不精确关联的权重。同时，confidence scorer 在关键词匹配不够精确时（matchCount 低、maxScore 低）会触发 hybrid 或 AI 模式，由 AI 提供更精准的建议。

**观察 2：冷启动场景的渐进覆盖。** "买东西"与"买菜"无法通过 char_jaccard 匹配（0.25 < 0.5），但用户首次手动创建子任务后，learn_log 会建立新的关键词路径。这种"用户行为驱动的渐进覆盖"是系统的核心设计——不追求首次完美匹配，而是通过使用逐步积累。

### 7.6 改进优先级

| 优先级 | 改进 | 类型 | 预期收益 |
|-------|------|------|---------|
| P0 | B: 组合式时间词模式 | 正则模式 (通用) | 消除遗漏时间词类噪声 |
| P1 | A: Rule 4b 结构性抑制 | 结构规则 (通用) | 消除边界合并冗余 |

P0 + P1 均为通用结构性改进，不引入任何硬编码词汇或领域假设。

## 附录 A: 算法伪代码

```
function extractKeywords(title):
  segments = Intl.Segmenter.segment(title)

  // Pass 1: Classify
  for each segment:
    if single Chinese char → append to current run
    else if temporal word → skip
    else → content word (end current run if any)

  // Pass 2: Adjacent joins
  for consecutive content words a, b:
    if len(a) >= 2 and len(b) >= 2 and len(a+b) <= 6:
      add a+b

  // Pass 3: Run recovery
  for each run:
    split at functional chars → sub-runs
    for each sub-run:
      generate bigrams, trigrams
      filter noise n-grams

  // Pass 4: Boundary merges
  for each run:
    if run.length == 1 and preceded by content word:
      merge content_word + run[0]         // 4a: 数据+库→数据库
    if run.length >= 2 and followed by content word:
      merge run[-1] + content_word        // 4b: 菜+市场→菜市场
    for recovered bigrams:
      merge bigram + following content     // 4c: 开题+报告→开题报告

  // Assemble with priority, prioritize known keywords
  return top 12
```
